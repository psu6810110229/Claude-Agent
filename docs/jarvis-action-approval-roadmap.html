J.A.R.V.I.S Approval OS Plan

แผนแก้ action, approval และ backlog UI ให้พร้อมขยายความสามารถของ Jarvis

เอกสารนี้อธิบายว่าจะปรับระบบจาก “AI เสนอ action แล้วผู้ใช้กด approve” ให้กลายเป็น control room ที่อ่านง่าย ปลอดภัย และรองรับ action เพิ่มขึ้นในอนาคตโดยไม่ทำให้ backend หรือ UI กระจัดกระจาย

เป้าหมายหลัก Jarvis ต้องพูดตรงกับสิ่งที่ backend ทำจริง เช่น done ไม่ควรกลายเป็น archive แบบคลุมเครือ
ขอบเขต safety AI ยัง propose-only เหมือนเดิม ทุก write สำคัญยังผ่าน approval queue
ผลลัพธ์ UX หน้า Approvals เป็น backlog board แบบ scan ง่าย พร้อม card/action copy ที่มนุษย์อ่านรู้เรื่อง

ปัญหาปัจจุบัน

สิ่งที่ทำให้ผู้ใช้สับสนจากเคส reminder ที่เลยกำหนด

Data model

Reminder ไม่มีสถานะ done

ระบบมีแค่ active/archived ทำให้คำว่า “เสร็จสิ้น” ถูก map เป็น archive ซึ่งไม่ตรงความรู้สึกของผู้ใช้

AI context

Jarvis ไม่รู้ approval ล่าสุด

หลัง approve แล้ว รอบถัดไป Claude เห็น history ว่าเคยสร้าง approval แต่ไม่เห็นสถานะล่าสุดว่า approved/rejected

UI copy

Action label ยังเป็นภาษา backend

ข้อความแบบ reminder.archive หรือ “เก็บถาวรไหม” ไม่เหมาะกับงานที่ผู้ใช้ตั้งใจให้ “เสร็จแล้ว”

โครงสร้างใหม่ที่ควรวาง

แยก domain, action, approval และ UI copy ให้ชัดก่อนเพิ่ม capability ใหม่

1 Domain State เพิ่มสถานะที่สะท้อนชีวิตจริง เช่น reminder done แยกจาก archived
2 Action Registry รวม schema, label, risk, executor และ UI copy ของ action ไว้ที่เดียว
3 Approval Policy กำหนดว่า action ไหนต้อง approve, action ไหน read-only, action ไหน forbidden
4 Execution Context ส่งสถานะ approval ล่าสุดให้ Jarvis รู้ ไม่ให้ตอบเดาจาก history
5 Backlog Board หน้า Approvals แยก Pending/Approved/Rejected เป็น column เหมือน GitHub backlog

Fallback & uncertainty messages

Jarvis ควรพูดตรงๆ ในแชทเมื่อทำไม่ได้ ทำได้บางส่วน หรือยังไม่แน่ใจ โดยไม่ทำให้เข้มงวดจนไม่กล้ารายงาน

Cannot do

ทำไม่ได้หรือยังไม่มี capability

ถ้าระบบยังไม่มีสิทธิ์หรือ capability นั้น Jarvis ต้องบอกตรงๆ และเสนอทางเลือกที่ทำได้ตอนนี้

ตอนนี้ผมยังแก้ไข Google Calendar event เดิมไม่ได้ครับ ทำได้เฉพาะสร้าง event ใหม่ผ่าน approval เท่านั้น ถ้าต้องการ ผมช่วยร่าง event ใหม่ให้ตรวจได้
Uncertain

ทำแล้วแต่ยังไม่แน่ใจ

ถ้า backend ทำสำเร็จแต่ข้อมูลปลายทางหรือ context อาจยังไม่ sync ควรบอกสถานะจริงและบอกว่าควรเช็กอะไร

ผมส่งคำขออนุมัติแล้วครับ แต่ยังไม่ถือว่าทำเสร็จจนกว่าคุณจะกด Approve ถ้าเป็นเรื่องสำคัญ แนะนำให้เช็กหน้า Approvals อีกครั้ง
Partial

ทำได้บางส่วน

ถ้าทำได้บางรายการและบางรายการต้องรอข้อมูลเพิ่ม ให้แยกผลลัพธ์ชัดเจน ไม่พูดเหมารวมว่าทำเสร็จทั้งหมด

ผมปิด reminder ได้ 2 รายการแล้ว แต่อีก 1 รายการผมหา id ที่ตรงกันไม่เจอ ช่วยยืนยันชื่อรายการนั้นอีกครั้งได้ไหมครับ
Good behavior

หลักการตอบที่ควรใช้

  • บอกสถานะจริง: proposed, pending approval, approved, executed, failed
  • ถ้าไม่แน่ใจ ให้บอกระดับความแน่ใจและสิ่งที่ต้องเช็กต่อ
  • ถ้าทำไม่ได้ ให้เสนอ next best action ที่ปลอดภัย
  • ใช้ข้อความในแชทเป็นหลัก ไม่ซ่อนทุกอย่างไว้ใน error banner
Balance

อย่า strict จน Jarvis ไม่กล้ารายงาน

  • ตอบข้อมูล read-only ได้แม้ไม่มั่นใจ 100% ถ้าบอกข้อจำกัดชัดเจน
  • ถาม clarification เฉพาะเมื่อจำเป็นต่อ write/action ที่เสี่ยง
  • อย่าบล็อก summary หรือ insight เพียงเพราะมีข้อมูลบางส่วนขาด
  • ห้ามพูดว่า “ทำแล้ว” ถ้ายังเป็นแค่ pending approval

UX backlog coverage

รายการ UX ที่อนุมัติให้ค้นคว้าและเพิ่มในแผน โดยไม่เพิ่มซ้ำกับหัวข้อที่มีอยู่แล้ว

Approvals board แบบ backlog ในแผนแล้ว

ใช้ Pending / Approved / Rejected columns เหมือน board layout เพื่อให้ scan งานรออนุมัติได้เร็วขึ้น

Action wording แบบมนุษย์ ในแผนแล้ว

Action display registry จะเปลี่ยน action_type ให้เป็น title, summary, risk และปุ่มที่คนอ่านเข้าใจ

Fallback message ในแชท ในแผนแล้ว

มี fallback contract สำหรับทำไม่ได้ ไม่แน่ใจ ทำได้บางส่วน และ pending approval

สถานะ action หลัง approve ในแผนแล้ว

Recent approval context จะส่งสถานะ approved/rejected/executed ล่าสุดให้ Jarvis ใช้ตอบรอบถัดไป

แยก Done กับ Archived ในแผนแล้ว

Reminder done/completed แยกจาก archived เพื่อไม่ให้ “เสร็จสิ้น” ถูกตีความเป็น “เก็บถาวร”

Approval card details แบบ expand ในแผนแล้ว

Approvals board จะซ่อน raw JSON ไว้หลัง Details เพื่อให้หน้าไม่รกแต่ยัง debug ได้

Inline approval เปลี่ยนสถานะทันที ในแผนแล้ว

หลัง approve/reject card ควรเปลี่ยน badge และย้าย column หรือ update inline ทันที

Thinking state ที่บอกว่ากำลังทำอะไร เพิ่มในแผน

Thinking text ควรสะท้อน step จริง เช่น Checking reminders, Reading calendar, Preparing approval ไม่ใช่แค่ “...”

Streaming-like reply เพิ่มในแผน

ถ้ายังไม่ทำ streaming จริง ให้ใช้ staged reveal/typing animation เพื่อให้ผู้ใช้รู้ว่าระบบกำลังตอบ ไม่ใช่ค้าง

Better empty states เพิ่มในแผน

แต่ละหน้าไม่มีข้อมูลควรบอก next action ชัด เช่น “ยังไม่มี approval รออยู่” และปุ่มไปเริ่ม chat หรือสร้าง task

Consistent toast language เพิ่มในแผน

ใช้ชุดคำเดียวกันทั้งระบบ: Pending approval, Approved, Rejected, Done, Edited, Failed

Activity log แปลเป็นภาษาคน เพิ่มในแผน

แปลง chat.message.replied / approval.approve เป็นข้อความอ่านง่าย พร้อม filter ตาม Chat, Approval, Task, Calendar

Clarification UI เพิ่มในแผน

ถ้า Jarvis ต้องถามเพิ่ม ให้เป็น clarification card พร้อมปุ่ม/field ตอบต่อ ไม่ใช่แค่ข้อความยาวใน bubble

Better session handling เพิ่มในแผน

แสดงสถานะ session ปัจจุบัน เวลาเริ่ม และ confirm dialog ที่สวย/ปลอดภัยก่อน archive session

Message grouping เพิ่มในแผน

ข้อความต่อเนื่องจากคนเดียวกันควร group กัน ลด timestamp ซ้ำ และทำให้ chat อ่านลื่นขึ้น

Markdown/rendering ที่ครบขึ้น เพิ่มในแผน

รองรับ list, bold, italic, underline, code, links แบบ safe rendering เพื่อไม่ให้เห็น raw markdown

Source/context icon เพิ่มในแผน

คำตอบควรมี chip/icon เล็กๆ เช่น Calendar, Reminders, Tasks, Memory เพื่อบอกว่า Jarvis อ้างอิงจากอะไร

ภาพหน้า Approvals ใหม่

เปลี่ยนจาก list ยาวเป็น board ที่ scan ได้เร็ว

แผน Sprint

ทำทีละชั้น เพื่อลดความเสี่ยงและ test ได้ง่าย

1

Reminder done/completed

เพิ่มสถานะ done ให้ reminders และทำให้ “ตั้งเป็นเสร็จสิ้น” ไม่ต้อง archive อีกต่อไป

  • เพิ่ม schema/status validation
  • เพิ่ม repository update status
  • ปรับ executor ให้รองรับ done
  • เพิ่ม smoke/focused test ถ้ามี surface backend เปลี่ยน
2

Recent approval context

ให้ Jarvis เห็น approval ล่าสุดแบบ compact เพื่อไม่ตอบว่า “อาจยังค้างอยู่” หลังผู้ใช้ approve แล้ว

  • เพิ่ม query recent approvals
  • ส่ง id/action/status/summary เข้าคอนเท็กซ์
  • ปรับ prompt ให้ใช้สถานะนี้เป็นแหล่งอ้างอิง
3

Fallback message contract

กำหนดวิธีพูดของ Jarvis เมื่อทำไม่ได้ ทำได้บางส่วน รอ approval หรือไม่มั่นใจพอ

  • เพิ่ม prompt rules สำหรับ status language: proposed, pending, executed, failed
  • ปรับ error/failure path ให้ตอบในแชทแบบช่วยผู้ใช้ ไม่ใช่มีแต่ banner
  • กำหนด threshold: read/report ได้พร้อม caveat แต่ write/action ต้องชัดก่อนเสนอ
4

Action display registry

สร้าง registry สำหรับข้อความที่ UI ใช้แสดง action เพื่อให้ Home inline approval และ Approvals page ใช้ logic เดียวกัน

  • title/summary/risk badge
  • approve/reject button labels
  • fallback สำหรับ action ที่ยังไม่มี formatter
5

Conversation polish pass

ปรับประสบการณ์ chat ให้รู้สึกเหมือนผู้ช่วยที่กำลังทำงานจริงและอ่านง่ายขึ้น

  • Thinking state แสดง step เช่น Reading calendar / Checking reminders / Preparing approval
  • Streaming-like reply หรือ staged reveal สำหรับคำตอบยาว
  • Message grouping และ markdown safe rendering ที่รองรับ list/code/link
  • Source/context chips สำหรับ Calendar, Tasks, Reminders, Memory
  • Consistent toast language และ better empty states ในหน้าหลัก
6

Approvals backlog board

ปรับหน้า Approvals เป็น Pending / Approved / Rejected columns พร้อม card ที่อ่านง่าย

  • column scroll แยกกัน
  • raw JSON อยู่หลัง Details
  • approve/reject แล้ว card ย้าย column ทันที
  • Activity log แปล event key เป็นภาษาคน และเชื่อมกลับไปยัง approval/card ที่เกี่ยวข้อง
7

Capability-ready foundation

เตรียมโครงไว้สำหรับอนาคตที่ Jarvis ทำงานได้มากขึ้น โดยยังไม่เพิ่ม connector ใหม่

  • แยก capability manifest จาก prompt hardcode
  • กำหนด risk level และ approval policy
  • ทำเอกสาร action lifecycle: proposed -> pending -> approved/rejected -> executed/failed
Expected outcome

ผลลัพธ์ที่ควรได้

  • Jarvis ตอบตรงกับ state จริงของระบบ
  • ผู้ใช้เห็นชัดว่า action ไหนรอ approve, action ไหนทำแล้ว
  • เมื่อทำไม่ได้หรือไม่แน่ใจ Jarvis บอกตรงๆ ในแชทพร้อมทางเลือกถัดไป
  • คำว่า done, archived, rejected, executed ไม่ปนกัน
  • หน้า Approvals กลายเป็น control room สำหรับความปลอดภัย
  • Chat อ่านลื่นขึ้นด้วย grouped messages, richer markdown และ source/context chips
Watch-outs

ข้อควรระวัง

  • อย่าเพิ่ม direct write route ที่ bypass approval queue
  • อย่าเพิ่ม Google Calendar update/delete โดยไม่ได้อนุมัติ scope ใหม่
  • migration ต้องระวังข้อมูล reminder เดิม
  • context ให้ AI ต้อง compact ไม่ dump database หรือ memory contents
  • อย่าทำ fallback ให้ระวังเกินไปจน Jarvis ไม่กล้าสรุปข้อมูล read-only ที่มี caveat ได้
